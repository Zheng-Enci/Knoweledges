# 文档高亮与强调规范 🎨

> 本文档规定技术文档中各种强调方式的使用场景，确保文档风格统一、重点突出

我们在写技术文档时，经常需要强调某些内容。但加粗、高亮、斜体这么多方式，到底什么时候用哪个？如果滥用，文档会显得花里胡哨；如果不用，重点又不够突出。所以我们需要一套统一的规范，让每篇文档的强调方式都保持一致。

## 1. 强调方式总览 📋

> 本章介绍文档中可用的所有强调方式及其适用场景

我们的技术文档支持以下几种强调方式，每种都有明确的用途：

| 强调方式 | 语法示例 | 视觉效果 | 使用频率 |
|---------|---------|---------|---------|
| 加粗 | `**文字**` | 文字变粗 | 最常用 |
| 斜体 | `*文字*` | 文字倾斜 | 较少用 |
| 行内代码 | `` `代码` `` | 等宽字体+灰色背景 | 技术术语 |
| 黄色背景高亮 | `<mark>文字</mark>` | 黄色背景 | 特别注意 |
| emoji 标记 | `🔴 **文字**` | 彩色圆点+加粗 | 分类标记 |
| 引用块 | `> 文字` | 左侧竖线+灰色背景 | 提示说明 |

**核心原则**：一种强调方式对应一种语义，不要混用。如果一段文字既加粗又用高亮，读者会困惑——这到底是"重要"还是"特别注意"？

---

## 2. 加粗的使用规范 💪

> 本章详细说明加粗的使用场景和禁忌

加粗是我们最常用的强调方式，但也不是随便用的。以下是明确的使用场景：

### 2.1 什么时候用加粗 ✅

**（1）首次出现的专业术语**

当文档中第一次出现某个专业术语时，用加粗标注，帮助读者识别关键词。

> 示例：**缩放点积注意力**（Scaled Dot-Product Attention）是 Transformer 架构中最核心的注意力计算方式。

**（2）核心概念和关键定义**

对文档主题至关重要的概念，需要加粗以突出其重要性。

> 示例：注意力机制的核心思想是**加权求和**。

**（3）步骤和阶段名称**

流程中的关键步骤、阶段名称，用加粗让读者快速抓住结构。

> 示例：**第一步**：生成 Q、K、V 矩阵。**第二步**：计算注意力分数并缩放。

**（4）重要结论和总结性语句**

段落或章节的结论性内容，加粗以提示读者这是重点。

> 示例：**这个公式是整个 Transformer 的基石。**

### 2.2 什么时候不要用加粗 ❌

**（1）不要整段加粗**

如果一整段都是加粗，等于没有强调。加粗的价值在于"对比"，周围都是普通文字，中间几个加粗的词才显眼。

**（2）不要连续多个句子加粗**

连续加粗会稀释强调效果。如果一句话里超过 3 个加粗词，建议重新组织句子，只保留最关键的。

**（3）不要在标题中额外加粗**

标题本身已经通过字号和格式突出了，不需要再加粗。标题中的加粗是多余的。

## 3. 黄色背景高亮（mark标签）的使用规范 🟨

> 本章说明 mark 标签的适用场景

`<mark>` 标签会在文字上添加黄色背景，比普通加粗更醒目。因为它的视觉冲击力强，所以要克制使用。

### 3.1 什么时候用 mark ✅

**（1）需要读者特别注意的内容**

容易忽略但很重要的信息，用 mark 标签高亮。

> 示例：<mark>注意：除以 √d_k 是为了防止点积值过大导致 Softmax 梯度消失。</mark>

**（2）警告和易错点**

常见错误、陷阱、副作用等，用 mark 标签警示读者。

> 示例：<mark>警告：不要在生产环境中直接打印 API 密钥。</mark>

**（3）关键参数和数值**

文档中提到的关键参数、阈值、默认值等，用 mark 高亮方便查找。

> 示例：学习率建议设置为 <mark>0.001</mark>，批次大小建议设置为 <mark>32</mark>。

### 3.2 什么时候不要用 mark ❌

**（1）不要大面积使用**

如果一段文字超过 20 个字都用 mark，黄色背景会显得很刺眼，而且失去强调的意义。

**（2）不要和普通加粗混用**

`<mark>**文字**</mark>` 这种组合过于强烈，除非是极其重要的警告，否则不要叠加使用。

**（3）不要在代码块中使用**

代码块已经有自己的语法高亮了，再加 mark 会干扰代码阅读。

---

## 4. emoji 标记的使用规范 🎭

> 本章说明如何用 emoji 做分类标记

emoji 标记是我们文档的特色之一，用彩色圆点配合加粗文字，形成一套"颜色语义系统"。这种方式兼容性好，在 VS Code 预览中也能正常显示。

### 4.1 emoji 标记规范

| emoji | 含义 | 使用场景 |
|-------|------|---------|
| 🔴 | 红色警告 | 关键概念、重要警告、易错点 |
| 🔵 | 蓝色术语 | 技术术语、专业名词、定义 |
| 🟢 | 绿色推荐 | 正确做法、推荐方案、最佳实践 |
| 🟡 | 黄色注意 | 需要注意的细节、副作用 |
| ⚠️ | 警告 | 严重警告、可能导致错误的行为 |
| 💡 | 提示 | 补充说明、小技巧、背景知识 |

### 4.2 使用示例

> 🔴 **红色警告**：表示关键概念或重要警告
> 🔵 **蓝色术语**：表示技术术语或定义
> 🟢 **绿色推荐**：表示正确行为或推荐做法
> 💡 **提示**：这是一个补充说明

### 4.3 使用原则

**（1）统一格式**：emoji 后面必须跟一个空格，然后是加粗文字。

**（2）不要自创语义**：上面的表格已经定义了每种 emoji 的含义，不要混用或自创。

**（3）列表中使用**：emoji 标记最适合用在引用块（`>`）或列表中，作为分类提示。

## 5. 行内代码的使用规范 💻

> 本章说明行内代码的适用场景

行内代码（用反引号包裹）不仅用于代码片段，也是技术文档中标记特殊内容的重要方式。

### 5.1 什么时候用行内代码 ✅

**（1）代码片段和命令**

> 示例：使用 `pip install transformers` 安装依赖。

**（2）技术术语和专有名词**

函数名、类名、文件名、配置项等，用行内代码标记。

> 示例：`Attention` 类实现了缩放点积注意力机制。

**（3）参数和变量名**

> 示例：`d_k` 表示 Query 和 Key 的维度。

**（4）文件路径和 URL**

> 示例：配置文件位于 `~/.bashrc`。

### 5.2 使用原则

**（1）不要过度使用**

如果一句话里出现 3 个以上的行内代码，建议改为普通文字描述，只在最关键的地方用行内代码。

**（2）不要在标题中使用**

标题中的行内代码会破坏标题的视觉层次，建议标题用普通文字，正文中再用行内代码解释。

---

## 6. 引用块的使用规范 📢

> 本章说明引用块（blockquote）的适用场景

引用块（以 `>` 开头）在技术文档中常用于提示、警告、补充说明等场景。

### 6.1 什么时候用引用块 ✅

**（1）提示和补充说明**

> 💡 提示：如果你对矩阵的基本运算还不熟悉，可以先看看相关章节。

**（2）重要警告**

> ⚠️ **警告**：修改此配置可能导致服务重启，请在维护窗口期操作。

**（3）引用外部内容**

> 引用：根据官方文档，该函数在 v2.0 版本后已废弃。

### 6.2 使用原则

**（1）引用块内可以叠加其他强调方式**

引用块本身是一种"容器"，内部可以使用加粗、mark、emoji 等强调方式。

**（2）不要嵌套引用块**

引用块嵌套会让层级混乱，如果需要多层提示，改用列表或分段。

---

## 7. 斜体的使用规范 📝

> 本章说明斜体的适用场景

斜体在我们的文档中使用频率较低，主要用于以下场景：

**（1）外来语和英文术语**

> 示例：*Transformer* 模型在 *NLP* 领域广泛应用。

**（2）书名、论文名**

> 示例：推荐阅读论文 *Attention Is All You Need*。

**（3）语气强调**

> 示例：这*不是*正确的做法。

**原则**：斜体不要和加粗混用（`* **文字** *`），这种组合视觉效果差，且语义不明确。

---

## 8. 组合使用规范 🎯

> 本章说明多种强调方式的组合规则

### 8.1 推荐的组合方式

| 组合 | 示例 | 使用场景 |
|------|------|---------|
| 引用块 + emoji + 加粗 | `> 🔴 **警告**：内容` | 分类提示 |
| 引用块 + mark | `> <mark>注意：内容</mark>` | 特别提醒 |
| 加粗 + 行内代码 | `**`d_k`**` | 强调参数 |

### 8.2 禁止的组合方式

| 组合 | 原因 |
|------|------|
| 加粗 + 斜体 | 语义冲突，视觉效果差 |
| mark + 加粗 | 过于强烈，除非极其重要 |
| 行内代码 + mark | 破坏代码可读性 |
| 整段 mark | 视觉刺眼，失去强调意义 |

---

## 9. 总结 🏁

> 本章总结文档高亮与强调的核心原则

**一句话原则**：强调方式是为"语义"服务的，不是为了"好看"。

| 强调方式 | 核心语义 | 使用频率 |
|---------|---------|---------|
| 加粗 | 重要、关键 | 高 |
| mark | 特别注意、警告 | 中 |
| emoji + 加粗 | 分类标记 | 中 |
| 行内代码 | 技术术语、代码 | 高 |
| 引用块 | 提示、警告容器 | 中 |
| 斜体 | 外来语、语气强调 | 低 |

**检查清单**：

- [ ] 加粗是否只用于真正重要的内容？
- [ ] mark 标签是否没有大面积使用？
- [ ] emoji 标记是否遵循了语义规范？
- [ ] 是否避免了禁止的组合方式？
- [ ] 整篇文档的强调风格是否统一？
