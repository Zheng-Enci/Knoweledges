# 文档编写示例 📚

本文档展示如何按照本项目的规则编写一篇规范的技术文档。通过阅读本文，你将了解文档的完整结构、写作风格、格式要求等各个方面。建议所有参与文档编写的作者先阅读本文，确保输出内容符合统一标准。

## 章节阅读路线图 🗺️

```mermaid
flowchart LR
    A["1. 文档命名规范"] --> B["2. 文档结构要求"] --> C["3. 写作风格指南"] --> D["4. 格式与高亮规范"] --> E["5. 内容组织技巧"]
```

我们先从文档命名开始，这是写作的第一步。名字起好了，读者才能一眼知道这篇文档讲什么。接着我们会看文档的整体结构，包括章节怎么安排、流程图怎么画。然后进入写作风格部分，这是让文档读起来舒服的关键。之后是格式和高亮规范，教你如何用视觉元素突出重点。最后分享一些内容组织的实用技巧，帮你把技术内容表达得更清晰。

## 1. 文档命名规范 📝

### 1.1 编号规则 🔢

文档文件名必须有编号，编号从01开始递增：

- `01-文档名.md`
- `02-文档名.md`
- `03-文档名.md`

编号必须出现在文件名和主标题最前面，格式为 `01-`、`02-` 等。

### 1.2 文件名与主标题一致 📋

文档的主标题必须与文件名**完全一致**，包括编号。比如文件名叫 `03-注意力机制基础.md`，主标题就必须是：

```markdown
# 03-注意力机制基础 📚
```

不能省略编号，也不能改动文字。这样读者从文件名就能知道文档内容，从标题也能对应到文件。

### 1.3 名称要具体明确 🎯

文档名要准确反映内容边界，避免范围过广。比如：

| 不好的命名 | 问题 | 好的命名 |
|-----------|------|---------|
| `03-深度学习.md` | 范围太广 | `03-梯度下降优化器.md` |
| `05-注意力机制.md` | 包含太多内容 | `05-缩放点积注意力代码实现.md` |

好的文档名让读者一眼能看出这是什么内容，边界在哪里。

## 2. 文档结构要求 🏗️

### 2.1 章节流程图 📊

当文档包含多个主章节时，必须在文档开头添加 Mermaid 流程图，展示主章节的阅读顺序和知识脉络。

流程图只包含主章节标题，用单向箭头 `-->` 表示阅读顺序。流程图下方要有文字说明，解释为什么是这种顺序。

### 2.2 章节层次 📐

- 一级标题：文档主标题（`# 标题`）
- 二级标题：主章节（`## 1. 标题 📚`）
- 三级标题：子章节（`### 1.1 标题 🔧`）

第一章概述使用一段话，不要分点。章节之间必须有衔接过渡，说明前后章节的逻辑关系。

## 3. 写作风格指南 ✍️

### 3.1 第一人称叙述 🗣️

必须使用第一人称方式叙述，如"我们将学习"、"我们需要掌握"、"我们需要理解"。禁止使用第三人称客观陈述。

> 🔵 **示例**：
> - ✅ 我们将学习注意力机制的核心概念
> - ❌ 本文介绍了注意力机制的核心概念

### 3.2 口语化表达 💬

文档必须口语化，像朋友聊天一样自然。多用段落式叙述，减少分点列表。技术概念要用通俗易懂的方式说明，适当使用类比和比喻。

> 🔵 **示例**：
> - ✅ 你可以把正则表达式想象成一个超级聪明的搜索助手
> - ❌ 正则表达式是一种用于匹配字符串中字符组合的模式

### 3.3 有趣生动 🎨

内容要引人入胜，避免枯燥的技术堆砌。表达个人理解和体会，让读者感受到你在和他交流。

## 4. 格式与高亮规范 🎨

### 4.1 强调方式 📋

| 方式 | 语法 | 用途 |
|------|------|------|
| 加粗 | `**文字**` | 重要概念、关键术语 |
| 行内代码 | `` `代码` `` | 函数名、参数、命令 |
| mark高亮 | `<mark>文字</mark>` | 警告、易错点、关键参数 |
| emoji标记 | `🔴 **文字**` | 分类提示 |
| 引用块 | `> 文字` | 提示、警告容器 |

### 4.2 emoji 语义 🎭

| emoji | 含义 | 示例 |
|-------|------|------|
| 🔴 | 警告/关键概念 | 🔴 **注意**：除以 √d_k 防止梯度消失 |
| 🔵 | 术语/定义 | 🔵 **缩放点积注意力**：Transformer核心机制 |
| 🟢 | 推荐/正确做法 | 🟢 **建议**：学习率设置为 0.001 |
| ⚠️ | 严重警告 | ⚠️ **危险**：修改配置将导致服务重启 |
| 💡 | 提示/补充 | 💡 提示：先阅读矩阵运算基础 |

### 4.3 使用原则 ⚡

- **加粗**：首次出现的术语、核心概念、步骤名称、重要结论
- **mark**：容易忽略的信息、常见错误、关键数值。不要大面积使用，不要和加粗叠加
- **行内代码**：函数名、类名、参数、文件路径。一句话不超过 3 个
- **引用块**：作为容器包裹提示/警告，内部可叠加 emoji 或 mark
- **禁止组合**：加粗+斜体、mark+加粗、行内代码+mark、整段mark

### 4.4 表情符号位置 📍

- 每个章节标题必须包含表情符号
- 每个二级和三级标题必须包含表情符号
- 关键概念和重要提示必须包含表情符号
- 适度使用，避免过度

## 5. 内容组织技巧 🧰

### 5.1 章节过渡 🌉

章节之间必须有衔接过渡，说明前后章节的逻辑关系。常用句式：

- "接下来我们将学习..."
- "前面我们了解了...，现在深入探讨..."
- "掌握了...之后，我们来看看..."

### 5.2 代码示例 💻

代码示例必须考虑跨平台兼容性，优先选择 Windows/Linux/Mac 都支持的方案。代码块使用正确的语法高亮：

```python
import torch

# 检查CUDA是否可用
print(torch.cuda.is_available())
```

### 5.3 表格使用 📊

使用表格对比数据，增强可读性：

| 符号 | 含义 | 示例 |
|------|------|------|
| `*` | 出现0次或多次 | `ca*t` 匹配 "ct"、"cat" |
| `+` | 出现1次或多次 | `ca+t` 匹配 "cat"、"caat" |
| `?` | 出现0次或1次 | `ca?t` 匹配 "ct"、"cat" |

### 5.4 参考资料 📚

在文档中引用搜索结果时，必须在引用处（内容紧跟其后）添加参考资料链接：

单个链接：
```markdown
- [标题-平台名](URL)
```

多个链接：
```markdown
内容段落...

（此处填写具体知识点名称）参考资料：
- [标题1 -- 平台名](URL1)
- [标题2 -- 平台名](URL2)
```

### 5.5 最后更新时间 📅

每篇文档末尾必须包含文档最后更新时间：

```markdown
---

**最后更新时间：2026-05-03**
```
