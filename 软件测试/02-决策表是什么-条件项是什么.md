## 1. 决策表概述 📚 / Overview of Decision Tables

> 📖 **Note:** 本章介绍决策表的本质、适用场景和核心价值 / This chapter introduces the nature, applicable scenarios, and core value of decision tables.

### 1.1 什么是决策表？ 🤔 / What is a Decision Table?

**决策表（Decision Table）**，也称**判定表**，是一种以表格形式系统化表达**输入条件组合**与**输出动作**之间逻辑关系的测试设计工具。它源于需求工程中的决策表技术，被广泛应用于软件黑盒测试领域。

**核心公式** 💡：

> 决策表 = 条件组合 × 动作映射

即：将问题的所有输入条件进行排列组合，为每一种有效的组合指定相应的预期动作。

**一个直观的结构** 🧱：

```
┌──────────────────────────────────┐
│         条件桩 (Condition Stub)   │  ← 列出所有输入条件
├──────────────────────────────────┤
│  条件项 (Condition Items)         │  ← 每个条件取值的组合
├──────────────────────────────────┤
│         动作桩 (Action Stub)      │  ← 列出所有可能动作
├──────────────────────────────────┤
│  动作项 (Action Items)           │  ← 每个组合对应的动作
└──────────────────────────────────┘
         ↕
     每列 = 一条规则
```

### 1.2 为什么需要决策表？ 🎯 / Why Decision Tables?

在实际软件测试中，很多业务逻辑并不简单——一个功能的正确行为往往取决于**多个条件的组合作用**。例如：

- 一个电商平台有**会员等级**（普通/金牌/钻石）、**订单金额**、**是否节日促销**等多个条件
- 折扣规则因这些条件的组合而变化
- 仅靠等价类划分或边界值分析无法系统地覆盖所有组合

**决策表的三大核心价值** 💪：

1. **完备性** ✅：自动生成所有有效条件组合，不会遗漏任何逻辑分支
2. **严谨性** 🔒：基于"强组合等价类"思想，遵循**独立性假设**——输入变量彼此独立互不影响
3. **可化简性** ✂️：通过合并相同动作的规则，消除冗余用例，降低测试成本

### 1.3 适用场景 🎯 / When to Use Decision Tables?

| 适用场景 | 不适用场景 |
|---------|-----------|
| 条件之间存在逻辑组合关系 | 条件之间不存在相关性 |
| 动作依赖于不同条件取值 | 只有单个条件影响输出 |
| 需要覆盖所有有效组合 | 需要处理无效等价类 |
| 业务规则复杂且多变 | 输入变量之间存在强依赖/互斥 |

> 💡 **Key Insight**：决策表不处理无效等价类，也不需要遵循单缺陷假设——它专注于覆盖**所有有效输入组合**，是最完整的功能测试方法之一。

### 1.4 决策表 vs 其他测试方法 ⚔️ / Comparison with Other Methods

| 方法 | 核心思想 | 覆盖程度 | 冗余 |
|------|---------|---------|------|
| 等价类划分 | 输入数据分组取代表性值 | 低 | 低 |
| 边界值分析 | 边界附近最易出错 | 中 | 低 |
| **决策表法** | **所有条件组合** | **高（完备）** | **可化简** |
| 正交试验法 | 用正交表选代表性组合 | 中高 | 低 |

## 2. 决策表核心组成 🧱 / Core Components of a Decision Table

> 📖 **Note:** 本章详细介绍决策表的结构和核心构成要素 / This chapter details the structure and core components of a decision table.

一个完整的决策表由**五大要素**组成，构成了一个二维表格结构。

### 2.1 条件桩 (Condition Stub) 📌

**条件桩**位于决策表的左上方，列出问题的**所有输入条件**。条件是影响决策的各种因素，通常是二值判断（是/否、真/假、0/1）或多值枚举。

**条件桩的特点**：
- 每个条件必须是**原子性**的（不可再拆分的独立判断）
- 条件之间应满足**独立性假设**（互不影响）
- 一般情况下条件的先后次序无关紧要

**示例**：为一个登录功能设计决策表，条件桩可能包括：
- 用户名是否有效？
- 密码是否正确？
- 账号是否被锁定？

### 2.2 条件项 (Condition Item) 🔢

**条件项**是针对条件桩的具体取值。在有限条目决策表中，每个条件可取 **True（Y）** 或 **False（N）** 两个值；在扩展条目决策表中，每个条件可取多个值。

> 💡 **条件项将在第3章详细讲解，这里先了解其在结构中的位置。**

### 2.3 动作桩 (Action Stub) 🎬

**动作桩**位于条件桩的下方，列出在特定条件组合下可能采取的**所有操作或输出**。

**动作桩的特点**：
- 列出所有可能的输出结果（不论是否对应到每个规则）
- 动作之间不要求互斥——同一条规则下可能执行多个动作
- 每个动作是原子性的操作描述

**示例**（接上例）：
- 允许登录
- 提示"用户名不存在"
- 提示"密码错误"
- 提示"账号已被锁定"
- 记录登录失败次数

### 2.4 动作项 (Action Item) ✅

**动作项**是在条件项各组取值下，系统实际采取的动作标记。通常用 **Y（执行）** 和 **N（不执行）** 表示该动作是否被触发。

> 💡 **动作项将在第4章详细讲解。**

### 2.5 规则 (Rule) 📏

**规则**是决策表中**贯穿条件项和动作项的一列**。每一条规则代表一个具体的条件取值组合及其对应的动作。

一个具有 n 个条件的有限条目决策表，最多包含 **2^n** 条规则（即所有条件的完全组合数）。

### 2.6 整体结构图示 📊 / Structural Diagram

```
┌──────────────────────┬──────┬──────┬──────┬──────┐
│      条件桩          │  规则1 │  规则2 │  规则3 │  规则4 │
├──────────────────────┼──────┼──────┼──────┼──────┤
│ C1: 条件1？          │  Y   │  Y   │  N   │  N   │  ← 条件项
│ C2: 条件2？          │  Y   │  N   │  Y   │  N   │  ← 条件项
├──────────────────────┼──────┼──────┼──────┼──────┤
│      动作桩          │      │      │      │      │
├──────────────────────┼──────┼──────┼──────┼──────┤
│ A1: 动作1            │  Y   │  N   │  N   │  Y   │  ← 动作项
│ A2: 动作2            │  N   │  Y   │  Y   │  N   │  ← 动作项
└──────────────────────┴──────┴──────┴──────┴──────┘
```

## 3. 条件项详解 🔍 / Detailed Explanation of Condition Items

> 📖 **Note:** 本章深入讲解条件项——决策表中最核心、最关键的构成要素 / This chapter provides an in-depth explanation of condition items — the most critical component of a decision table.

### 3.1 什么是条件项？ 🤔 / What is a Condition Item?

**条件项（Condition Item）** 是条件桩（Condition Stub）在特定规则下的具体取值。如果把决策表比作一个"逻辑对照表"，条件项就是表中列出"每个条件在当前规则下是真是假"的那些单元格。

**通俗理解** 🍕：想象你在点披萨

- **条件桩**：你下单需要决定的几个因素——"要不要辣？"、"要不要芝士？"、"要不要肉？"
- **条件项**：针对具体一份订单，你对每个因素的回答——"辣=要"、"芝士=不要"、"肉=要"

条件项的组合决定了这份订单最终会得到什么披萨（动作）。

### 3.2 条件项的取值方式 🎯 / Value Types

**有限条目（Limited Entry）** 🔲：

每个条件只有两个取值，通常记作：

| 符号 | 含义 | 别名 |
|------|------|------|
| **Y** | 条件成立（True） | Yes, 1, T |
| **N** | 条件不成立（False） | No, 0, F |

示例（有限条目）：

| 条件 | 规则1 | 规则2 | 规则3 | 规则4 |
|------|-------|-------|-------|-------|
| 金额 > 100？ | Y | Y | N | N |
| 是会员？ | Y | N | Y | N |

**扩展条目（Extended Entry）** 🎨：

每个条件可取多个离散值，如枚举值、类别或范围：

| 条件 | 规则1 | 规则2 | 规则3 |
|------|-------|-------|-------|
| 支付方式 | 信用卡 | 支付宝 | 微信 |
| 会员等级 | 金牌 | 银牌 | 普通 |

> 💡 当条件的取值类型是连续的数值范围时（如"年龄 0-150"），应先在**等价类划分**的基础上确定若干个离散区间，再用扩展条目的方式填入决策表。

### 3.3 条件项的排列规律 📐 / Arrangement Pattern

在有限条目决策表中，条件项的排列有着清晰的二进制模式。假设有 n 个条件，规则总数为 2^n，条件项的排列遵循以下规律：

- **第1个条件**（C1）：前 2^(n-1) 个规则为 Y，后 2^(n-1) 个规则为 N
- **第2个条件**（C2）：每段 2^(n-2) 个规则交替 Y/N
- **第 i 个条件**（Ci）：每段 2^(n-i) 个规则交替 Y/N

**示例**：n = 3 个条件，共 8 条规则

| 条件 | 规则1 | 规则2 | 规则3 | 规则4 | 规则5 | 规则6 | 规则7 | 规则8 |
|------|-------|-------|-------|-------|-------|-------|-------|-------|
| C1   | Y     | Y     | Y     | Y     | N     | N     | N     | N     |
| C2   | Y     | Y     | N     | N     | Y     | Y     | N     | N     |
| C3   | Y     | N     | Y     | N     | Y     | N     | Y     | N     |

> 💡 这种规律保证所有条件组合都被覆盖到，且不遗漏也不重复。它等价于**二进制计数器**：将 Y 视为 1、N 视为 0，每条规则就是一个二进制数从大到小（或从小到大）排列。

### 3.4 条件项的语义解读 📖 / Semantic Interpretation

条件项在决策表中的含义不仅仅是"Y/N 两种取值"，它背后代表了**测试用例的输入数据**。

| 条件项 Y/N | 测试含义 | 示例 |
|-----------|---------|------|
| C = Y | 条件 C 满足/成立/有效 | 金额 > 100，用户名存在 |
| C = N | 条件 C 不满足/不成立/无效 | 金额 ≤ 100，用户名不存在 |

条件项的组合本质上定义了一个**测试数据域**。例如：
- C1=Y, C2=Y, C3=N → 构造一条测试数据，让条件1和条件2成立、条件3不成立
- 这条测试数据验证的是"当条件为 Y/Y/N 时，系统是否执行预期的动作"

## 4. 动作项详解 ⚡ / Detailed Explanation of Action Items

> 📖 **Note:** 本章讲解动作项的含义、填写规则和常见模式 / This chapter explains the meaning, filling rules, and common patterns of action items.

### 4.1 什么是动作项？ 🤔 / What is an Action Item?

**动作项（Action Item）** 是动作桩（Action Stub）在特定规则下的取值标记。它指示在某个条件组合下，系统是否会执行某个特定动作。

**通俗理解** 📋：继续披萨的例子
- **动作桩**：披萨店可能执行的操作——"加辣"、"加芝士"、"加肉"
- **动作项**：针对你点的订单，是否执行某个操作——规则1下"加辣=是"、"加芝士=否"、"加肉=是"

### 4.2 动作项的填写规则 📝 / Filling Rules

**规则1：每个动作必须显式标记**

在每条规则下，每个动作都应有明确的标记，不允许空白。

**规则2：Y = 执行，N = 不执行**

| 标记 | 含义 |
|------|------|
| **Y (Yes)** | 在该条件组合下，此动作会被触发执行 |
| **N (No)** | 在该条件组合下，此动作不会被触发 |

**规则3：同一规则下允许多个动作同时为 Y**

这反映了真实业务——系统可能同时做多件事。例如登录成功时，既要"允许登录"也要"重置登录失败次数"。

### 4.3 动作项的模式类型 🔄 / Pattern Types

**互斥模式** ❌：同一规则下只有一个 Action 被触发
```
| 动作       | 规则1 | 规则2 | 规则3 |
|------------|-------|-------|-------|
| 允许登录   |  Y    |  N    |  N    |
| 拒绝登录   |  N    |  Y    |  Y    |
```

**并行模式** 🔗：同一规则下多个 Action 同时被触发
```
| 动作              | 规则1 | 规则2 |
|-------------------|-------|-------|
| 应用折扣          |  Y    |  N    |
| 赠送积分          |  Y    |  N    |
| 免运费            |  N    |  Y    |
```

**顺序模式** ⏩：动作有先后顺序（决策表本身不表达时序，但在测试脚本中需注意执行顺序）

## 5. 决策表的类型 📋 / Types of Decision Tables

> 📖 **Note:** 本章介绍决策表的三种类型及其适用场景 / This chapter introduces three types of decision tables and their applicable scenarios.

### 5.1 有限条目决策表 🔲 / Limited Entry Decision Table

所有条件桩只有两个取值（Y/N），所有动作桩也只有两个取值（执行/不执行）。

**特点**：
- 结构简单、规则数量明确（2^n）
- 适用于业务规则清晰、条件呈二值判断的场景
- 最常见的决策表形式

**示例**：登录验证的有限条目决策表

| 条件/动作 | R1 | R2 | R3 | R4 |
|-----------|----|----|----|----|
| C1: 用户名存在？ | Y | Y | N | N |
| C2: 密码正确？ | Y | N | Y | N |
| A1: 允许登录 | Y | N | N | N |
| A2: 提示密码错误 | N | Y | N | N |
| A3: 提示用户不存在 | N | N | Y | Y |

> 💡 上面这个例子中，R3（用户名不存在但密码"正确"）在现实中是逻辑矛盾的——用户名不存在何来密码正确？这类**不可能组合**在决策表化简时会被消除。

### 5.2 扩展条目决策表 🎨 / Extended Entry Decision Table

条件桩可取多个值（超过两个），动作桩也可取多个结果值。

**特点**：
- 规则数量不再遵循 2^n 公式，而是各条件取值个数的笛卡尔积
- 适用于条件为枚举值、类别或离散区间的场景
- 表格可能更大更复杂

**示例**：运费计算

| 条件/动作 | R1 | R2 | R3 | R4 | R5 | R6 |
|-----------|----|----|----|----|----|----|
| C1: 配送区域 | 同城 | 同城 | 省内 | 省内 | 跨省 | 跨省 |
| C2: 是否加急 | 是 | 否 | 是 | 否 | 是 | 否 |
| A1: 运费金额 | ¥10 | ¥5 | ¥20 | ¥10 | ¥40 | ¥20 |

### 5.3 混合条目决策表 🔀 / Mixed Entry Decision Table

部分条件使用有限条目（Y/N），部分条件使用扩展条目（多值）。

**适用场景**：当条件集合中既有二值判断也有多值枚举时使用。

## 6. 构建决策表 🏗️ / Building a Decision Table

> 📖 **Note:** 本章介绍构建决策表的标准五步法以及规则化简技巧 / This chapter introduces the standard five-step method for building decision tables and rule simplification techniques.

### 6.1 五步构建法 📐 / Five-Step Construction Method

**步骤1：列出所有条件桩和动作桩** 📋

从需求规格说明书中提取：
- 所有影响决策的**输入条件** → 条件桩
- 所有可能的**操作输出** → 动作桩

**步骤2：确定规则数量** 🔢

- 有限条目：n 个条件 → 2^n 条规则
- 扩展条目：各条件取值数的乘积

**步骤3：填入条件项** ✏️

按照二进制排列规律（第3章3.3节），填写每个规则下的条件取值。

**步骤4：填入动作项** ✅

对每一条规则（每一列），根据业务逻辑判断应执行哪些动作，在对应动作行的位置填写 Y/N。

**步骤5：化简（合并规则）** ✂️

将动作项相同的相邻规则进行合并，消除冗余。

### 6.2 规则化简的两种情形 ✂️ / Rule Simplification

**情形1：动作完全相同的规则合并**

当两条或多条规则的动作项完全相同，且条件项只在某一个（或多个）条件上有差异时，可以合为一条规则。差异条件用"无关"标记（通常记作 `—` 或 `*`）。

**示例**：

化简前：

| 条件/动作 | R1 | R2 |
|-----------|----|----|
| C1: 金额 > 100？ | Y | Y |
| C2: 是会员？ | Y | N |
| A1: 应用折扣 | Y | Y |

化简后（C2 成为无关条件）：

| 条件/动作 | R1 |
|-----------|----|
| C1: 金额 > 100？ | Y |
| C2: 是会员？ | — |
| A1: 应用折扣 | Y |

**情形2：不可能组合的删除**

某些条件组合在现实业务中不可能出现。例如"用户名为空但密码正确"——用户名不存在就无法验证密码，这类规则应删除。

### 6.3 化简后的测试用例设计 🧪 / Test Case Design After Simplification

化简后的决策表中，每条规则设计一个测试用例即可。这样做的好处：
- 消除了冗余的测试用例
- 保持了逻辑覆盖的完备性
- 降低了测试执行成本

## 7. 完整示例 🎯 / Complete Example

> 📖 **Note:** 本章通过一个完整的实战案例，展示从需求到决策表的全过程 / This chapter demonstrates the complete process from requirements to decision table through a practical case.

**需求**：一个电影购票系统，对购票用户施加以下折扣规则：

1. 如果是**会员**（Condition 1），票价享受 8 折优惠
2. 如果购买**场次是首映**（Condition 2），需加收 10 元附加费
3. 如果购买**张数 ≥ 3**（Condition 3），享受 9 折优惠
4. 所有折扣可叠加（会员 8 折 × 张数 9 折）

### 步骤1：列出条件桩和动作桩 📋

**条件桩**（3 个条件 → 2^3 = 8 条规则）：
- C1: 是会员？
- C2: 首映场次？
- C3: 张数 ≥ 3？

**动作桩**：
- A1: 享受 8 折会员折扣
- A2: 加收 ¥10 附加费
- A3: 享受 9 折张数折扣
- A4: 无折扣（原价）

### 步骤2-3：填入条件项 ✏️

| 条件/动作 | R1 | R2 | R3 | R4 | R5 | R6 | R7 | R8 |
|-----------|----|----|----|----|----|----|----|----|
| C1: 会员？ | Y | Y | Y | Y | N | N | N | N |
| C2: 首映？ | Y | Y | N | N | Y | Y | N | N |
| C3: ≥3张？ | Y | N | Y | N | Y | N | Y | N |

### 步骤4：填入动作项 ✅

| 条件/动作 | R1 | R2 | R3 | R4 | R5 | R6 | R7 | R8 |
|-----------|----|----|----|----|----|----|----|----|
| C1: 会员？ | Y | Y | Y | Y | N | N | N | N |
| C2: 首映？ | Y | Y | N | N | Y | Y | N | N |
| C3: ≥3张？ | Y | N | Y | N | Y | N | Y | N |
| A1: 会员8折 | Y | Y | Y | Y | N | N | N | N |
| A2: 加收¥10 | Y | Y | N | N | Y | Y | N | N |
| A3: 张数9折 | Y | N | Y | N | Y | N | Y | N |
| A4: 无折扣 | N | N | N | N | N | N | N | Y |

**解读** 🔍：
- R1（会员 + 首映 + ≥3张）：全触发，享受 8 折 × 9 折叠加 + 加收 ¥10
- R4（会员 + 非首映 + <3张）：仅享受 8 折
- R8（非会员 + 非首映 + <3张）：无任何折扣 → 原价

### 步骤5：化简决策表 ✂️

观察 R5-R7（非会员）：它们的动作项并不完全相同，无法合并。

观察 R1-R4（会员）：A2（加收¥10）在 R3、R4 中为 N，无法简单合并。

检查是否存在"不可能组合"？在这个例子中所有组合都是可能的，因此不删除规则。

最终化简后保持 8 条规则不变。

### 测试用例映射 🧪

| 用例编号 | 规则 | 会员 | 首映 | 张数 | 预期结果 |
|---------|------|------|------|------|---------|
| TC1 | R1 | 是 | 是 | 3张 | 8折×9折 + ¥10 |
| TC2 | R2 | 是 | 是 | 1张 | 8折 + ¥10 |
| TC3 | R3 | 是 | 否 | 3张 | 8折×9折 |
| TC4 | R4 | 是 | 否 | 1张 | 8折 |
| TC5 | R5 | 否 | 是 | 3张 | 9折 + ¥10 |
| TC6 | R6 | 否 | 是 | 1张 | + ¥10 |
| TC7 | R7 | 否 | 否 | 3张 | 9折 |
| TC8 | R8 | 否 | 否 | 1张 | 原价 |

## 8. 总结 📝 / Summary

本章介绍了决策表的完整知识体系，核心要点回顾 🎯：

| 要素 | 作用 | 关键点 |
|------|------|--------|
| **条件桩** 📌 | 列出所有输入条件 | 原子性、独立性 |
| **条件项** 🔢 | 指定各规则下的条件取值 | 二进制排列（有限条目）、笛卡尔积（扩展条目） |
| **动作桩** 🎬 | 列出所有可能动作 | 完整性、原子性 |
| **动作项** ✅ | 标记各规则下执行的动作 | 每个动作显式标记 |
| **规则** 📏 | 条件组合 + 动作映射 | 贯穿条件项和动作项的列 |

**核心要点** 💡：

- 决策表是**最严谨**的功能测试方法，保证了输入条件组合的**完备性** 🔒
- **条件项**是决策表的核心——它定义了每一个可能的数据输入场景，是测试用例设计的源头 🎯
- 有限条目决策表的条件项排列遵循二进制规律：n 个条件 → 2^n 条规则 📐
- **化简**是决策表的重要步骤，通过合并相同动作的规则消除冗余测试用例 ✂️
- 决策表不处理无效等价类，专注于有效输入域上的完整覆盖 ✅
