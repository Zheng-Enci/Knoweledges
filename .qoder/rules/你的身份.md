**不要假设。不要隐藏困惑。要明确表达。**

在实现之前：
- 明确陈述你的假设。如果不确定，就去问。
- 如果存在多种解释，都要提出——不要默默选择。
- 如果存在更简单的方法，请指出。在合理的情况下要敢于反对。
- 如果有不清楚的地方，停下来。说出困惑所在。去问。

## 2. 简单优先

**用最少的代码解决问题。不添加任何推测性的内容。**

- 不要添加需求之外的功能。
- 不要为只使用一次的代码创建抽象。
- 不要添加没有要求的"灵活性"或"可配置性"。
- 不要处理不可能出现的情况的错误处理。

问自己："一位资深工程师会说这过度复杂吗？"如果会，就简化。

## 3. 精确修改

**只触碰必须修改的部分。只清理自己的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有坏的东西。
- 匹配现有风格，即使你用不同方式来做。
- 如果你注意到无关的死代码，提出来——不要删除。

当你的更改产生孤立内容时：
- 移除因你的更改而变得未使用的导入/变量/函数。
- 不要删除原本就存在的死代码，除非被要求。

检验标准：每一行更改都应该直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让测试通过"
- "修复 bug" → "编写能复现问题的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，简要说明计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强的成功标准让你可以独立循环。弱的标准（"让它能工作"）需要不断澄清。

# 调试日志规范

## 强制要求

**AI 编写的代码必须包含详细的 console.log 调试日志。**

## 规则详情

每次编写新代码时：

1. **必须添加 console.log**，无论现有代码是否有调试日志
2. **说明日志位置**：注释说明该日志在哪个函数/模块中
3. **说明输出内容**：注释说明该日志输出了什么数据
4. **关键节点必记录**：
   - 函数入口
   - 关键判断分支
   - API 调用前后
   - 数据转换处理
   - 错误捕获

## 日志格式

### 标准格式
```javascript
// [DEBUG] 函数名/模块：描述 + 关键数据
console.log('[functionName] 描述信息:', data)
```

### 错误日志
```javascript
// [ERROR] 函数名/模块：错误描述
console.error('[functionName] 错误信息:', error)
```

## 示例

```javascript
/**
 * 获取学生信息
 */
async function getStudentInfo(studentId) {
  // [DEBUG] getStudentInfo：函数入口，记录传入参数
  console.log('[getStudentInfo] 开始获取学生信息, studentId:', studentId)
  
  try {
    const response = await fetch(`/api/students/${studentId}`)
    
    // [DEBUG] getStudentInfo：API 响应，记录原始返回数据
    console.log('[getStudentInfo] API 响应数据:', response.data)
    
    if (!response.ok) {
      // [ERROR] getStudentInfo：HTTP 错误，记录状态码
      console.error('[getStudentInfo] API 请求失败, status:', response.status)
      throw new Error(`HTTP error! status: ${response.status}`)
    }
    
    // [DEBUG] getStudentInfo：成功获取，记录关键数据
    console.log('[getStudentInfo] 学生信息获取成功:', response.data.studentName)
    return response.data
  } catch (error) {
    // [ERROR] getStudentInfo：异常捕获，记录错误详情
    console.error('[getStudentInfo] 获取学生信息异常:', error.message)
    throw error
  }
}
```

## 重要说明

- ✅ 即使其他代码没有 console.log，新写的代码也**必须**有
- ✅ 调试日志不是可选的，是**强制要求**
- ✅ 每个关键节点都应该有对应的日志
- ✅ 使用统一的前缀格式 `[函数名]` 便于过滤和追踪
- 🧹 后期可通过脚本统一清理（如 `scripts/remove_console_logs.py`）

## 清理方式

生产环境发布前，运行以下脚本移除所有 console.log：

```bash
python scripts/remove_console_logs.py
```
