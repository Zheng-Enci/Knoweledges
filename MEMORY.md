# MEMORY.md

本文件存储用户的长期决策与偏好设置，包括：

- 用户的核心偏好和习惯
- 重要决策和选择
- 系统配置信息
- 个人化的工作流程

## 内容专注性要求

- **文档内容必须严格围绕标题主题**：所有章节和内容都应直接服务于文档标题的核心主题
- **删除与标题无关的内容**：定期检查并移除偏离主题的章节或段落
- **保持内容聚焦**：避免添加与主题无关的扩展内容，确保文档结构紧凑
- **主题一致性**：确保各章节内容相互关联，共同支撑文档的核心主题

## 用户的核心偏好和习惯

### 写作风格
- 要简洁，不要啰嗦
- **必须使用第一人称方式叙述**（如"我们将学习"、"我们需要掌握"、"我们需要理解"），禁止使用第三人称客观陈述
- 表达个人理解和体会
- 技术概念要用通俗易懂的方式说明，适当使用类比和比喻
- **文档中必须包含丰富的表情符号**：
  - **每个章节标题必须包含表情符号**（如"## 1. 概述 📚"）
  - **每个二级和三级标题必须包含表情符号**
  - **每个列表项必须包含表情符号**
  - **每个表格标题必须包含表情符号**
  - **关键概念和重要提示必须包含表情符号**
  - **段落中的重要句子可以添加表情符号**
  - **代码示例的说明文字可以添加表情符号**
  - **增强可读性和亲和力，鼓励适度多用**

### 文档结构
- 第一章概述使用一段话，不要分点
- 章节要有清晰的层次结构
- 所有标题必须有序号，包括文档一级标题（如 `# 1. 标题`）、二级标题（`## 1.1 标题`）、三级标题（`### 1.1.1 标题`）
- 不需要单独的目录章节
- 使用表格对比数据
- 流程类内容必须使用Mermaid图表可视化

### 格式规范
- 标题使用规范的Markdown格式
- 列表使用标准"- "格式，不要使用"|- "
- 代码块使用正确的语法高亮
- **文档文件名必须有编号**（如 01-文档名.md），编号从01开始递增
- Mermaid流程图中的步骤也要有编号
- 首次出现的专业术语要用括号包含解释（如 LLM（Large Language Model，大语言模型））
- 保持文档格式统一和规范

### 内容要求
- 先联网搜索学习相关内容，再编写文档
- 确保内容准确性和时效性
- 添加实际案例和应用场景
- 使用表格、图表等增强可读性
- 根据实际情况调整章节结构
- **代码示例必须考虑跨平台兼容性**：优先选择 Windows/Linux/Mac 都支持的方案，避免使用特定平台才能运行的代码（如 Detectron2 在 Windows 上安装复杂，应选用 EfficientDet 等替代方案）
- **章节之间必须有衔接过渡**：在章节末尾或开头添加过渡段落，说明前后章节的逻辑关系、内容承接，帮助读者建立知识脉络（如"接下来我们将学习..."、"前面我们了解了...，现在深入探讨..."）
- **文档之间必须多互动联系**：
  - **每篇文档开头必须回顾上篇文档学习内容，并附上链接**（如"通过上篇文档[08c. 检索算法与策略-混合检索](https://juejin.cn/...)的学习，我们掌握了..."）
  - **文档内涉及其他文档知识点时，必须附上链接**（如"在[08a. 检索算法与策略-稠密检索技术](https://juejin.cn/...)中我们学习了..."）
  - 跨文档引用必须具体到章节，方便读者定位
  - 链接格式：掘金链接 / CSDN链接

### 沟通偏好
- 回复简洁直接，不添加不必要的解释
- 使用客观陈述，避免情感化表达
- 优先使用中文进行交流
- 不确定时主动搜索而非直接询问
- 检测到新偏好或重要决策时自动更新 MEMORY.md，无需询问

### 复杂内容处理
- 对于复杂的技术公式、算法原理等内容，采用"看看就好，会使用就行"的原则
- 在介绍复杂内容时，先展示完整内容，然后添加简化理解，强调核心思想和实际应用
- 避免让读者因复杂内容而产生畏难情绪，重点放在如何使用和实际应用上

### 文档编写规则
- 所有回答必须依据真实事实，不可以编造内容。如果遇到不知道的内容，直接回答不知道即可。
- 不可以直接复制他人的语句，要用自己的话重新表述内容。如果需要引用内容，必须按照MLA格式来标注。
- 写作风格要适中，介于正式书面学术风格和日常口语表达之间。
- 每一个句子都必须有主语，不使用复杂的长句子，尽量用简短的句子来写内容。
- 把所有不常用、生僻的词语，换成大家日常使用的普通词语。
- 把所有过渡词、连接词，换成最基础、最常用的简单词汇。
